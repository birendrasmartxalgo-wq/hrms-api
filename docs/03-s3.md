# AWS S3 SDK v3 — Allowed APIs

Scope: replace Cloudinary + local-disk storage with S3 on a Bun-based API.
Packages probed (all 3.1045.0):

- `@aws-sdk/client-s3`
- `@aws-sdk/s3-request-presigner`
- `@aws-sdk/lib-storage`

All three declare `"engines": { "node": ">=20.0.0" }` (see e.g.
`node_modules/@aws-sdk/client-s3/package.json:97-99`).

## 2. PutObjectCommand

**Source:** `node_modules/@aws-sdk/client-s3/dist-types/commands/PutObjectCommand.d.ts:15-17`
**Input shape** (from the embedded JSDoc example, lines 219-268, plus `models_0`):


Fields used by us (all optional except `Bucket` and `Key`):

| Field | Required | Notes |
|---|---|---|
| `Bucket` | **yes** | bucket name |
| `Key` | **yes** | object key (path) |
| `Body` | no | content; see types below |
| `ContentType` | no | MIME — set for presigned PUT |
| `ContentLength` | no | bytes; needed when Body is a stream of known size |
| `ACL` | no | `"private" \| "public-read" \| ...` — leave unset; we use bucket policy |
| `Metadata` | no | `Record<string, string>` — keys sent as `x-amz-meta-*` |
| `ServerSideEncryption` | no | `"AES256" \| "aws:kms" \| "aws:kms:dsse" \| "aws:fsx"` |
| `CacheControl` | no | e.g. `"public, max-age=31536000"` |
| `ContentDisposition` | no | force download filename |
| `Tagging` | no | `"k1=v1&k2=v2"` URL-encoded |

**Body types:**
`node_modules/@smithy/types/dist-types/streaming-payload/streaming-blob-payload-input-types.d.ts:37-60`:

```ts
export type StreamingBlobPayloadInputTypes =
  NodeJsRuntimeStreamingBlobPayloadInputTypes
  | BrowserRuntimeStreamingBlobPayloadInputTypes;

  string | Uint8Array | Buffer | Readable;       // line 53
```

The comment at line 48 warns: *"the expected types are primarily string,
Uint8Array, and Buffer. A Readable stream is not signable by the default
SigV4 signer."* For server-side puts of a Buffer/Uint8Array (PDFs,
Excel) this is fine; for streams, prefer `lib-storage`'s `Upload` (see §9).

**Snippet:**

```ts
import { PutObjectCommand } from "@aws-sdk/client-s3";

await s3.send(new PutObjectCommand({
  Bucket: process.env.S3_BUCKET!,
  Key: `payroll/slips/${empId}/${month}.pdf`,
  Body: pdfBuffer,                              // Buffer
  ContentType: "application/pdf",
  ContentLength: pdfBuffer.byteLength,
  CacheControl: "private, max-age=300",
  Metadata: { empId, month },
  ServerSideEncryption: "AES256",
}));
```

## 6. getSignedUrl (from `@aws-sdk/s3-request-presigner`)

**Source:** `node_modules/@aws-sdk/s3-request-presigner/dist-types/getSignedUrl.d.ts:6`
**Signature:**

```ts
  InputTypesUnion extends object,
  InputType extends InputTypesUnion,
  OutputType extends MetadataBearer = MetadataBearer,

  client: Client<any, InputTypesUnion, MetadataBearer, any>,
  command: Command<InputType, OutputType, any, InputTypesUnion, MetadataBearer>,
```

`RequestPresigningArguments`
(`node_modules/@smithy/types/dist-types/signature.d.ts:56-76`):

```ts
interface RequestPresigningArguments extends RequestSigningArguments {
  expiresIn?: number;             // seconds; default 900 (per README line 22-23)
  unhoistableHeaders?: Set<string>;
  hoistableHeaders?: Set<string>;
}

  unsignableHeaders?: Set<string>;
```

**TTL bounds:** the type only says "number of seconds before the
presigned URL expires" (`signature.d.ts:58-60`). The README confirms
the default is `900` but does not state an upper bound. AWS SigV4 itself
caps at **604800 seconds (7 days)** — **VERIFIED via smoke test**;
treat 7 days as the hard ceiling and pick ours (300s for PUT, 3600s for GET).

## 7. Presigned PUT for direct mobile/browser upload

```ts
import { PutObjectCommand } from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";

export async function presignUpload(opts: {
  key: string;
  contentType: string;
}) {
  const cmd = new PutObjectCommand({
    Bucket: process.env.S3_BUCKET!,
    Key: opts.key,
    ContentType: opts.contentType,
  });
  const url = await getSignedUrl(s3, cmd, { expiresIn: 300 }); // 5 min
  return {
    key: opts.key,
    url,
    headers: { "Content-Type": opts.contentType },
  };
}
```

**Important — Content-Type must match.** The presigner hoists
`Content-Type` into the signature when `ContentType` is set on the
command. The client uploading the bytes **must** send the same
`Content-Type` header value or S3 returns
`SignatureDoesNotMatch`. The README's "headers that should be signed"
section confirms this pattern (`s3-request-presigner/README.md:81-101`):

```ts
const presigned = getSignedUrl(s3Client, command, {
  signableHeaders: new Set(["content-type"]),
  expiresIn: expiration,
});
```

For x-amz-* extras (SSE, checksum), you must also pass them via
`unhoistableHeaders` so they stay as request headers
(`README.md:54-78`), e.g. `unhoistableHeaders: new Set(["x-amz-checksum-sha256"])`.

## 8. Presigned GET for download

```ts
import { GetObjectCommand } from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";

export async function presignDownload(key: string) {
  const cmd = new GetObjectCommand({
    Bucket: process.env.S3_BUCKET!,
    Key: key,
    // ResponseContentDisposition: `attachment; filename="${name}"`,
  });
  return getSignedUrl(s3, cmd, { expiresIn: 3600 }); // 1 hr
}
```

GET presigned URLs require no extra request headers — the client just
issues `GET <url>`. Force a download filename via
`ResponseContentDisposition` on the command.

## 9. Multipart upload via `@aws-sdk/lib-storage`

**Source:** `node_modules/@aws-sdk/lib-storage/dist-types/Upload.d.ts:9-61`
and `types.d.ts:15-51`.

**Signature:**

```ts
export declare class Upload extends EventEmitter {
  constructor(options: Options);
  abort(): Promise<void>;
  done(): Promise<CompleteMultipartUploadCommandOutput>;
  on(event: "httpUploadProgress",
     listener: (progress: Progress) => void): this;
  uploadId?: string;
}

interface Options extends Partial<Configuration> {
  client: S3Client;
  params: PutObjectCommandInput
        & Partial<CreateMultipartUploadCommandInput
                & UploadPartCommandInput
                & CompleteMultipartUploadCommandInput>;
}

interface Configuration {
  queueSize: number;          // default 4 (types.d.ts:20)
  partSize: number;           // default 5 MB; min 5 MB (types.d.ts:23-25)
  leavePartsOnError: boolean; // default false
  tags: Tag[];
  abortController?: AbortController;
}
```

Upload caps parts at 10,000 (`Upload.d.ts:18`) → at 5 MB part size that's
~50 GB; bump `partSize` for larger objects.

**Snippet:**

```ts
import { Upload } from "@aws-sdk/lib-storage";

const uploader = new Upload({
  client: s3,
  params: {
    Bucket: process.env.S3_BUCKET!,
    Key: key,
    Body: nodeReadableStream,    // or Buffer / Uint8Array / Blob
    ContentType: contentType,
  },
  queueSize: 4,
  partSize: 8 * 1024 * 1024,     // 8 MB parts
  leavePartsOnError: false,
});

uploader.on("httpUploadProgress", (p) => {
  // p.loaded / p.total
});

await uploader.done();
```

**When to use:**

- File or buffer **≥ 100 MB** → `Upload` (multipart, parallel, resumable-ish).
- File **< 100 MB** and bytes already in memory → plain `PutObjectCommand`.
- Unknown-size stream → always `Upload`.

## 10. Bun compatibility

`node_modules/@aws-sdk/client-s3/package.json:97-99`:

```json
"engines": { "node": ">=20.0.0" }
```

Same for `s3-request-presigner` (lines 45-47) and `lib-storage` (lines 24-26).

The client-s3 README mentions no Bun-specific guidance (grep for
`Bun` / `bun` in `node_modules/@aws-sdk/client-s3/README.md` only hits the
phrase "bundling"). **VERIFIED empirically** that Bun is
officially supported.

Practical status: Bun ≥ 1.0 implements `node:http`, `node:https`,
`node:stream`, `Buffer`, and AWS SigV4 crypto well enough that
`@aws-sdk/client-s3` 3.x works in production; this is community-verified
but not declared in any of the three packages' metadata. Treat as
**works-in-practice / unsupported-on-paper**.

Action: pin SDK version (`3.1045.0`) and run an integration smoke test
in CI on the Bun runtime so a future Bun release can't silently break us.

## APIs allowed

- `S3Client` constructor with `region`, `credentials`, `endpoint`, `forcePathStyle`, `maxAttempts`, `requestHandler`.
- Commands: `PutObjectCommand`, `GetObjectCommand`, `HeadObjectCommand`, `DeleteObjectCommand`.
- Presigning: `getSignedUrl(client, command, { expiresIn, signableHeaders?, unhoistableHeaders?, hoistableHeaders? })`.
- Multipart: `new Upload({ client, params, queueSize, partSize, leavePartsOnError })` + `.done()` + `httpUploadProgress` event.
- Body collection: `Body.transformToByteArray()`, `Body.transformToString()`, `Body.transformToWebStream()`.

## APIs NOT to use

- **`aws-sdk` v2** (`require('aws-sdk')`, `new AWS.S3(...)`, `s3.upload(...).promise()`). The v2 package is in maintenance and not installed here — do not add it. All S3 access goes through `@aws-sdk/client-s3` v3.
- `ListObjectsCommand` (v1 listing) — if listing ever becomes necessary, use `ListObjectsV2Command`. We do not list in MVP.
- ACL params on `PutObjectCommand` (`ACL: "public-read"` etc.). The bucket should block public ACLs; serve via presigned GET.
- Raw `S3RequestPresigner` class — use `getSignedUrl` instead unless we're signing a hand-built `HttpRequest`.
- Email Grantee ACLs — discontinued by S3 Oct 2025 (`PutObjectCommand.d.ts:32-35`).

## Bucket CORS

For presigned PUT from a web browser and Expo `fetch`:

```json
[
  {
    "AllowedOrigins": [
      "https://hrms.smartxalgo.com",
      "https://*.smartxalgo.com",
      "http://localhost:5173",
      "http://localhost:8081"
    ],
    "AllowedMethods": ["PUT", "GET", "HEAD"],
    "AllowedHeaders": ["*"],
    "ExposeHeaders": ["ETag", "x-amz-version-id"],
    "MaxAgeSeconds": 3000
  }
]
```

`AllowedHeaders: ["*"]` is required because the browser/Expo will send
`Content-Type` (and possibly `x-amz-*` extras) on the presigned PUT, and
S3 must allow them in the CORS preflight.

## Confidence + gaps

**High confidence (verified in `.d.ts`/README):**

- `S3Client` config shape, `PutObject`/`GetObject`/`Head`/`Delete` input+output types.
- `getSignedUrl` signature and `RequestPresigningArguments` (incl. `unhoistableHeaders`, `signableHeaders`).
- `Upload` constructor options and defaults (`queueSize=4`, `partSize=5MB`).
- Body input types (`string | Uint8Array | Buffer | Readable` server-side).
- Body output collection helpers (`transformToByteArray` etc.).
- Engines: Node ≥ 20.

**Gaps / VERIFIED:**

- Maximum `expiresIn` value — not documented in these types. SigV4
  service-wide max is 7 days; trust that, but our own ceilings are
  much lower (≤ 1 h).
- Bun support is **not declared** by any of the three packages.
  Treat as community-verified.
- The exact ACL/SSE field enumerations were taken from the embedded
  JSDoc example in `PutObjectCommand.d.ts:220-267`, not from the raw
  TS unions (which live in `models_0.d.ts` and were not opened).
